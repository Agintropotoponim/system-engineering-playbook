Введение

Цель документа

Настоящий документ определяет требования к программному обеспечению системы обработки городских обращений «24/7», предназначенной для приема, обработки, маршрутизации и контроля выполнения заявок от жителей города.

Система обеспечивает:

  • регистрацию обращений граждан;
  • автоматическую маршрутизацию заявок в соответствующие подрядные организации;
  • контроль сроков выполнения работ;
  • информирование заявителей о статусе обращений;
  • сбор аналитики для органов городской администрации;

ПО включает следующие компоненты:

  • клиентские приложения (мобильное и веб для граждан);
  • операторский интерфейс (call-центр / диспетчерская);
  • подсистема подрядчиков;
  • административная панель;
  • аналитический модуль;

Цель системы — обеспечить автономный прием заявок граждан и последующей маршрутизацией заявки в нужную подрядную организацию для устранения возникшей проблемы

Область применения

Система предназначена для использования органами местного самоуправления и муниципальными службами для управления обращениями граждан.

Основные сценарии применения:

  • управление городским хозяйством (ЖКХ, благоустройство);
  • обработка инцидентов городской инфраструктуры;
  • цифровизация взаимодействия «гражданин — власть»;
  • повышение прозрачности и контроля исполнения работ;

Система может применяться:

  • в городах любого масштаба;
  • в рамках программ «умный город» (Smart City);
  • в интеграции с государственными цифровыми платформами;

Общее описание

Перспектива продукта

Система состоит из нескольких основных компонентов:

  • Клиентское приложение (веб и мобильное), выполняющее:
    • создание и отправку обращений (с фото, описанием и геолокацией);
    • отображение статуса заявок;
    • получение уведомлений;

Серверная часть (Backend), выполняющая:

  • обработку и хранение заявок;
  • автоматическую маршрутизацию заявок;
  • контроль сроков выполнения (SLA);
  • защиту от спама и фейковых заявок;

ПО операторов (диспетчерская / call-центр):

  • просмотр и модерация обращений
  • ручная корректировка маршрутизации
  • связь с подрядчиками

ПО подрядчиков (исполнителей):

  • получение задач
  • обновление статуса выполнения
  • загрузка отчётов (фото/комментарии)

Административная и аналитическая система:

  • управление пользователями и ролями
  • просмотр статистики и отчетов
  • анализ проблемных зон

Основными функциями системы являются:

  • регистрация обращений с привязкой к карте
  • автоматическое распределение заявок
  • отслеживание статуса обращения в реальном времени
  • уведомление пользователей
  • контроль сроков выполнения (SLA)
  • сбор и визуализация аналитики

Ограничения

  • Система зависит от наличия интернета
  • Максимальное время обработки заявки системой — до нескольких секунд
  • Ограничение на размер загружаемых файлов (например, фото до 10 МБ)
  • Поддержка определённого количества одновременных пользователей
  • Зависимость от внешних сервисов (карты, уведомления)

Предположения и зависимости

  • У пользователей есть смартфон или компьютер с доступом в интернет
  • Подрядные организации подключены к системе
  • Используются сторонние сервисы:
    • карты и геокодирование
    • push/SMS/email уведомления
    • система авторизации
  • Пользователь указывает корректную информацию
  • Серверная инфраструктура доступна 24/7

Нефункциональные требования

Производительность

  • Время отклика системы при создании обращения: ≤ 2 секунд
  • Время загрузки списка обращений: ≤ 3 секунд
  • Поддержка одновременной работы не менее 10 000 пользователей
  • Время доставки уведомлений: ≤ 5 секунд
  • Обработка заявки сервером: ≤ 1 секунды

Надёжность

  • Система должна работать 24/7 с доступностью не менее 99,5%
  • Система должна сохранять все данные при сбоях
  • Система должна автоматически восстанавливаться после сбоев
  • Должно выполняться регулярное резервное копирование данных
  • При сбое внешних сервисов (например, карт) система должна продолжать работу в ограниченном режиме

Безопасность

  • Система должна поддерживать аутентификацию пользователей
  • Данные пользователей должны передаваться по защищённому протоколу (HTTPS)
  • Система должна ограничивать количество запросов (rate limiting)
  • Должна быть реализована защита от спама и фейковых заявок
  • Все действия пользователей должны логироваться (аудит)
  • Разграничение прав доступа по ролям

Удобство использования

  • Интерфейс должен быть интуитивно понятным
  • Процесс создания обращения должен занимать не более 1–2 минут
  • Интерфейс должен быть адаптирован под мобильные устройства
  • Система должна поддерживать несколько языков (например, русский и английский)
  • Все важные действия должны сопровождаться подсказками

Совместимость

  • Поддержка мобильных платформ: iOS и Android
  • Поддержка веб-браузеров: Chrome, Firefox, Safari (последние версии)
  • Интеграция с внешними сервисами:
    • карты (Яндекс.Карты)
    • уведомления (Уведомление в приложении, SMS, email)
  • Совместимость с государственными системами авторизации
  • Возможность масштабирования системы

Внешние интерфейсы

Интерфейсы пользователя (UI)

Интерфейс для граждан:

  • экран создания обращения (форма + карта + загрузка фото)
  • список обращений
  • экран статуса обращения
  • список уведомлений (о статусах заявки)
  • просмотр профиля

Интерфейс оператора:

  • список всех обращений с фильтрами
  • карта с отображением обращений
  • карточка обращения с возможностью редактирования
  • список подрядных организаций
  • просмотр профиля пользователя, который оставил заявку

Интерфейс подрядчика:

  • список задач
  • изменение статусов
  • загрузка фотоотчётов

Административная панель:

  • управление пользователями и ролями
  • аналитические дашборды
  • редактирование заявок и их просмотр

Программные интерфейсы

  • Backend (например, на Spring Boot / Node.js): REST API
  • WebSocket (или аналог): для обновлений в реальном времени
  • Сервисы карт (Яндекс.Карты API)
  • Сервисы уведомлений ( SMS-шлюзы, email)
  • Хранилище файлов (например, S3 или локальное)

Другие требования

  • Исходный код должен быть модульным
  • Покрытие кода unit-тестами не менее 70%
  • Обязательное наличие документации по установке и эксплуатации
  • Возможность масштабирования системы
  • Возможность обновления системы без остановки сервиса (rolling update)
  • Логгирование всех ошибок и событий
